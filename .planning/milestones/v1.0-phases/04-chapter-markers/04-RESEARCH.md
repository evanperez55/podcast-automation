# Phase 4: Chapter Markers - Research

**Researched:** 2026-03-17
**Domain:** ID3 chapter tags (mutagen), Podcast Namespace RSS chapters (JSON external file)
**Confidence:** HIGH

## Summary

The chapter data is already being generated by GPT-4 in `content_editor.py` — the `analyze_content()` method returns a `chapters` list with `start_timestamp` (HH:MM:SS), `title`, and `start_seconds` fields already parsed. Phase 4 is entirely about consuming that existing data in two new ways: (1) writing ID3 CHAP/CTOC frames into the final MP3, and (2) adding a `<podcast:chapters>` tag to the RSS feed that points to an external JSON file hosted alongside the episode.

The standard Python library for ID3 chapter frames is mutagen (currently 1.47.0). It supports CHAP and CTOC frames natively. The recommended RSS format is the Podcasting 2.0 `<podcast:chapters>` tag pointing to an external JSON file — Apple Podcasts (iOS 18+) and 31+ other apps support it. Writing both ID3 tags AND the JSON/RSS tag is the right approach: ID3 tags are embedded in the file (works offline, survives hosting changes), while the JSON/RSS tag works in podcast apps that don't process embedded ID3 metadata.

Apple Podcasts has specific requirements: at least 3 chapters, first chapter at 00:00:00, episodes longer than 10 minutes, chapters no shorter than 2 minutes, and titles no longer than 45 characters / 5 words. The GPT-4 prompt already requests 5-8 chapters with the first at 00:00:00, which satisfies this.

**Primary recommendation:** Create `chapter_generator.py` with two public methods: `embed_id3_chapters(mp3_path, chapters)` and `generate_chapters_json(chapters, episode_number, output_dir)`. Call `embed_id3_chapters` immediately after MP3 conversion (step 6). Call `generate_chapters_json` during RSS feed update (step 7.5) and store the resulting JSON path in episode data so `rss_feed_generator.py` can emit the `<podcast:chapters>` tag.

### Pattern 2: ID3 CHAP/CTOC Frame Writing
**What:** Write a CTOC (table of contents) frame and one CHAP frame per chapter into an existing MP3's ID3 tag.
**When to use:** After step 6 MP3 conversion, before step 7 Dropbox upload.
**Example:**
```python
# Source: https://mutagen.readthedocs.io/en/latest/user/id3.html
from mutagen.id3 import ID3, CTOC, CHAP, TIT2, CTOCFlags
from mutagen.mp3 import MP3

def embed_id3_chapters(self, mp3_path: str, chapters: list) -> bool:
    audio_info = MP3(mp3_path)
    total_duration_ms = int(audio_info.info.length * 1000)

    audio = ID3(mp3_path)
    # Remove any existing chapter frames before writing
    audio.delall("CHAP")
    audio.delall("CTOC")

    child_ids = [f"chp{i}" for i in range(len(chapters))]

    audio.add(CTOC(
        element_id="toc",
        flags=CTOCFlags.TOP_LEVEL | CTOCFlags.ORDERED,
        child_element_ids=child_ids,
        sub_frames=[TIT2(text=["Table of Contents"])],
    ))

    for i, chapter in enumerate(chapters):
        start_ms = int(chapter["start_seconds"] * 1000)
        if i + 1 < len(chapters):
            end_ms = int(chapters[i + 1]["start_seconds"] * 1000)
        else:
            end_ms = total_duration_ms

        audio.add(CHAP(
            element_id=f"chp{i}",
            start_time=start_ms,
            end_time=end_ms,
            sub_frames=[TIT2(text=[chapter["title"][:45]])],
        ))

    audio.save()
    return True
```

### Pattern 3: Podcast Namespace JSON + RSS Tag
**What:** Write a `chapters.json` file alongside the episode, then add the `<podcast:chapters>` tag to the RSS `<item>` element.
**When to use:** During step 7.5 RSS feed update.

**JSON format** (source: https://github.com/Podcastindex-org/podcast-namespace/blob/main/chapters/jsonChapters.md):
```json
{
  "version": "1.2.0",
  "chapters": [
    {"startTime": 0, "title": "Intro"},
    {"startTime": 330, "title": "We Talk About Lobsters"},
    {"startTime": 1200, "title": "Wrap Up"}
  ]
}
```

**RSS integration** (add to `<item>` and register namespace on `<rss>` root):
```xml
<!-- on <rss> root element, add: -->
xmlns:podcast="https://podcastindex.org/namespace/1.0"

<!-- inside each <item> that has chapters: -->
<podcast:chapters url="https://example.com/ep29_chapters.json"
                  type="application/json+chapters" />
```

**Python (mutagen-style, using ET)**:
```python
# Source: https://podcasting2.org/docs/podcast-namespace/tags/chapters
ET.register_namespace("podcast", "https://podcastindex.org/namespace/1.0")

chapters_elem = ET.SubElement(item, "podcast:chapters", {
    "url": chapters_url,
    "type": "application/json+chapters",
})
```

### Anti-Patterns to Avoid
- **Writing CHAP frames without CTOC:** Apple Podcasts requires a CTOC (table of contents) frame alongside the CHAP frames. Without CTOC, chapters may not display.
- **Not clearing existing CHAP/CTOC before writing:** If the pipeline runs twice on the same MP3, old chapter frames accumulate. Always call `audio.delall("CHAP")` and `audio.delall("CTOC")` first.
- **Titles over 45 characters:** Apple Podcasts truncates to 45 characters; slice at write time: `chapter["title"][:45]`.
- **Fewer than 3 chapters or chapters shorter than 2 minutes:** Apple Podcasts will not display chapters that don't meet these minimums. The GPT-4 prompt requests 5-8 chapters which satisfies the count minimum; warn if any individual chapter is shorter than 120 seconds.
- **Generating new chapters from scratch:** The `chapters` field already exists in the GPT-4 analysis output. Do not re-analyze the transcript; consume the existing data.

## Common Pitfalls

### Pitfall 1: CHAP end_time on Last Chapter
**What goes wrong:** The last CHAP frame needs an `end_time` equal to total audio duration. If set to 0 or left as the next chapter start, some players display the chapter as 0 seconds long.
**Why it happens:** There is no "next chapter" to derive end_time from for the final entry.
**How to avoid:** Load `MP3(mp3_path).info.length` (seconds), convert to ms, and use it as `end_time` for the last chapter.
**Warning signs:** Final chapter has zero or very short duration in mp3tag or mutagen read-back.

### Pitfall 2: ID3 Tag Does Not Exist Yet
**What goes wrong:** `ID3(mp3_path)` raises `ID3NoHeaderError` if the MP3 has no ID3 tag header at all.
**Why it happens:** The pipeline creates MP3s via ffmpeg which may not write an ID3 header when there are no tags.
**How to avoid:** Use `mutagen.id3.ID3` with a try/except and fall back to `mutagen.id3.ID3()` then `audio.save(mp3_path)` to create a new tag, or use `mutagen.File(mp3_path, easy=False)` which creates tags if absent.
**Warning signs:** `mutagen.id3._util.ID3NoHeaderError` in test or production logs.

Correct pattern:
```python
from mutagen.id3 import ID3, ID3NoHeaderError
try:
    audio = ID3(mp3_path)
except ID3NoHeaderError:
    audio = ID3()
    audio.save(mp3_path)
    audio = ID3(mp3_path)
```

### Pitfall 3: RSS Namespace Prefix Collision
**What goes wrong:** `ET.register_namespace("podcast", ...)` must be called before any `ET.parse()` or `ET.tostring()`, otherwise ElementTree uses auto-generated `ns0:` prefixes.
**Why it happens:** Python's ElementTree registers namespaces globally but loses them on import if not re-registered per process.
**How to avoid:** Add `ET.register_namespace("podcast", "https://podcastindex.org/namespace/1.0")` at module load time in `rss_feed_generator.py`, alongside the existing `itunes` and `content` registrations. Also add `xmlns:podcast` to the `<rss>` root element's attribute dict in `create_feed()`.
**Warning signs:** RSS output shows `<ns0:chapters ...>` instead of `<podcast:chapters ...>`.

### Pitfall 4: chapters JSON file needs a public URL
**What goes wrong:** The `<podcast:chapters url="...">` tag requires a publicly accessible URL. The JSON file exists locally during pipeline execution, but podcast apps fetch it at playback time.
**Why it happens:** The Podcast Namespace spec requires a URL, not a local path.
**How to avoid:** The JSON file should be uploaded to Dropbox alongside the MP3 (step 7), and the Dropbox shared link should be used as the URL. If the Dropbox shared link workflow is unavailable (test mode), the `<podcast:chapters>` tag should be omitted or use a placeholder. A simpler initial approach: store the JSON in `output/ep_N/` and add the URL to `episode_data` so the caller can inject it. If no URL is available, skip the RSS tag (don't break the feed).
**Warning signs:** Podcast app shows "unable to load chapters" or spinner that never resolves.

### Pitfall 5: Chapters data may be empty
**What goes wrong:** `content_editor.py` calls `analysis.setdefault("chapters", [])` — the chapters list could be empty if GPT-4 failed to produce it.
**Why it happens:** JSON parse failure, API error, or the LLM returned an empty array.
**How to avoid:** In `embed_id3_chapters` and `generate_chapters_json`, check `if not chapters: return` and log a warning. Don't fail the pipeline for missing chapters.

### Full embed_id3_chapters
```python
# Source: https://mutagen.readthedocs.io/en/latest/user/id3.html
from mutagen.id3 import ID3, ID3NoHeaderError, CTOC, CHAP, TIT2, CTOCFlags
from mutagen.mp3 import MP3

def embed_id3_chapters(self, mp3_path: str, chapters: list) -> bool:
    if not chapters:
        logger.warning("No chapters to embed in %s", mp3_path)
        return False

    # Get total duration for last chapter's end_time
    mp3_info = MP3(mp3_path)
    total_ms = int(mp3_info.info.length * 1000)

    try:
        audio = ID3(mp3_path)
    except ID3NoHeaderError:
        audio = ID3()
        audio.save(mp3_path)
        audio = ID3(mp3_path)

    audio.delall("CHAP")
    audio.delall("CTOC")

    child_ids = [f"chp{i}" for i in range(len(chapters))]
    audio.add(CTOC(
        element_id="toc",
        flags=CTOCFlags.TOP_LEVEL | CTOCFlags.ORDERED,
        child_element_ids=child_ids,
        sub_frames=[TIT2(text=["Chapters"])],
    ))

    for i, ch in enumerate(chapters):
        start_ms = int(ch["start_seconds"] * 1000)
        end_ms = int(chapters[i + 1]["start_seconds"] * 1000) if i + 1 < len(chapters) else total_ms
        title = ch["title"][:45]  # Apple Podcasts 45-char limit
        audio.add(CHAP(
            element_id=f"chp{i}",
            start_time=start_ms,
            end_time=end_ms,
            sub_frames=[TIT2(text=[title])],
        ))

    audio.save()
    logger.info("Embedded %d chapter(s) in %s", len(chapters), mp3_path)
    return True
```

### generate_chapters_json
```python
# Source: https://github.com/Podcastindex-org/podcast-namespace/blob/main/chapters/jsonChapters.md
import json

def generate_chapters_json(self, chapters: list, output_path: str) -> str:
    payload = {
        "version": "1.2.0",
        "chapters": [
            {"startTime": ch["start_seconds"], "title": ch["title"]}
            for ch in chapters
        ],
    }
    with open(output_path, "w", encoding="utf-8") as f:
        json.dump(payload, f, indent=2, ensure_ascii=False)
    return output_path
```

## Open Questions

1. **chapters.json hosting URL**
   - What we know: The `<podcast:chapters>` tag requires a public URL. The JSON file is generated locally during the pipeline. The project already uploads to Dropbox and obtains shared links.
   - What's unclear: Whether the Dropbox shared link for the JSON file will be available before the RSS feed update at step 7.5, and whether the pipeline's Dropbox link format is compatible (Dropbox `?dl=0` links vs. direct file links).
   - Recommendation: In the initial implementation, upload chapters.json to Dropbox alongside the MP3 in step 7 and pass the shared link into the RSS step. If no URL is available (test mode or Dropbox disabled), omit the `<podcast:chapters>` tag — don't break the feed.

2. **Fallback when chapters list is empty or only 1-2 items**
   - What we know: GPT-4 requests 5-8 chapters but could return fewer. Apple Podcasts requires at least 3.
   - What's unclear: Whether to fail silently or warn loudly when chapter count is below Apple's minimum.
   - Recommendation: Log a warning when `len(chapters) < 3` but still embed what we have. Do not skip ID3 embedding for 1 or 2 chapters — other players may accept them.
