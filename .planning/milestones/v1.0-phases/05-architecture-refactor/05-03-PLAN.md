Create pipeline/runner.py — the main orchestration module that replaces PodcastAutomation class.

**pipeline/runner.py must contain:**

1. `_init_components(test_mode, dry_run, auto_approve)` — Extract from PodcastAutomation.__init__(). Returns a components dict (see interfaces block from Plan 02). Include the dry_run shortcut (skip heavy init). Include _init_uploaders() as a module-level helper.

2. `run(args)` — The main entry point for episode processing. This replaces PodcastAutomation.process_episode(). It should:
   - Call _init_components() with flags from args
   - Build initial PipelineContext
   - Call each step function in order: run_ingest -> run_analysis -> run_audio (note: analysis uses transcript from audio step 2, but censor/normalize happen after analysis — read process_episode() carefully for the actual ordering)
   - Actually, the real order from process_episode() is: ingest -> transcribe (part of audio) -> analyze -> censor+normalize (part of audio) -> video -> distribute. Since audio.py wraps steps 2+4+4.5+6, and analysis is step 3 which runs BETWEEN transcribe and censor, you need to either: (a) split run_audio into run_transcribe and run_process_audio, or (b) have run_audio handle the interleaving internally by accepting a callback/flag. The SIMPLEST approach: have run_audio do transcribe first, return ctx, then run_analysis, then call a second audio function run_audio_post(ctx, components, state) for censor+normalize+mp3. OR: Keep it as one run_audio that does all audio steps in order (2, 4, 4.5, 6) and call run_analysis between the transcribe portion and the censor portion. RECOMMENDED: Split into `run_transcribe(ctx, components, state)` and `run_process_audio(ctx, components, state)` in audio.py, keeping `run_audio` as a convenience that calls both. Then in runner.py: run_ingest -> run_transcribe -> run_analysis -> run_process_audio -> run_video -> run_distribute. Update audio.py accordingly (this is the only change to a Plan 02 file — split the existing run_audio into two parts with run_audio calling both).

3. `dry_run(components)` — Extract from PodcastAutomation.dry_run_episode(). Validates pipeline wiring with mock data.

4. `run_with_notification(args, episode_number=None, dropbox_path=None, local_audio_path=None)` — Extract from _process_with_notification(). Wraps run() with Discord notification on success/failure.

5. `run_upload_scheduled()` — Extract from _run_upload_scheduled().

6. `run_analytics(episode_arg)` — Extract from _run_analytics().

7. `run_search(query)` — Extract from _run_search().

8. `list_episodes_by_number()` and `list_available_episodes()` — Extract from PodcastAutomation methods. These use DropboxHandler to list episodes.
